Collaborative Discussion 1: UML flowchart - Initial Post

OWASP (Open Web Application Security Project) Broken Access Control refers to a security vulnerability in web applications where restrictions on the proper access to sensitive data are not properly enforced (Bach-Nutman, 2020). This can result in unauthorized users gaining access to confidential information, modifying or deleting critical data, or performing unauthorized actions. (Bach-Nutman, 2020)

Broken Access Control often occurs when developers do not properly enforce access controls and authorization rules in their applications (Bach-Nutman, 2020). This can happen when they fail to properly validate user input, fail to properly implement authentication and authorization mechanisms, or when they rely on client-side controls instead of server-side controls (Bach-Nutman, 2020).

A real-world example of Broken Access Control can be seen in the 2019 data breach at Capital One (Madnick et al, 2020). The attacker was able to exploit a misconfigured firewall in the cloud infrastructure, which allowed them to gain access to the company's customer data, including names, addresses, credit scores, and more (Madnick et al, 2020). The attacker was able to access this sensitive information due to the lack of proper access controls and authentication mechanisms in place (Madnick et al, 2020).

To illustrate the OWASP Broken Access Control incident that occurred during the data breach at Capital One, I opted for a Use Case Diagram (Dobing & Parsons, 2006). A Use Case Diagram is a type of behavioural diagram that represents the interactions between actors and the system (Dobing & Parsons, 2006). In this case, the threat actor was a former Amazon Web Services employee, and the system was a rented cloud data server (Madnick et al, 2020).
