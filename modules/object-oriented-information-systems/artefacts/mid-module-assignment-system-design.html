Mid-Module Assignment: System Design

For this assignment, I designed a new software system to manage self-checkout at a supermarket chain. The system will enable customers to scan and weigh items and make payments using various methods. Supermarket staff can override transactions if needed, such as when a price has been scanned incorrectly or when they need to check a customer's age for age-restricted items. The system will also connect to the supermarket's warehouse systems to manage inventory.

Rationale

The class diagram shows the proposed object model for the self-service checkout system. The Product class contains information about the products such as name, price, barcode, and weight. This class is be used to track the products that are being scanned and added to the customer's shopping basket as well as the products in the warehouse stock. The ShoppingBasket class keeps track of the products added to the basket by the customer and the total cost. It also has methods for adding and removing items from the basket, as well as calculating the total cost of the items in the basket. The PaymentProcessor class handles the payment process and includes methods for processing different payment methods.

The PointOfServiceSystem class acts as the main class and coordinates the interaction between the other classes. The Customer class keeps track of customer information such as name and customer ID. The LoyaltyCard class is a sub-class of the Customer class, and it handles the calculation of loyalty point based discounts. The WarehouseSystem class keeps track of the products in stock and updates the stock levels when a purchase is made. The Receipt class generates a receipt for the customer at the end of the transaction. The Staff class contains information about staff members. More importantly, it contains methods for overriding transactions in order to remove products that have been scanned and to approve age verification restricted products.

Overall, the design of the system is based on the functionality and data requirements for the self-service checkout system. The classes and relationships are chosen to provide a logical and efficient way to track and manage the various aspects of the checkout process, including scanning products, calculating the total cost, handling payments, and updating inventory. The use of different classes and relationships allows for a clear separation of concerns and makes the system more modular and flexible, making it easier to add new functionality or make changes in the future.

One key choice made in the design is the use of the PointOfServiceSystem class as the main class to coordinate the interaction between the other classes. This allows for a clear and centralized control flow for the checkout process, making it easier to understand and maintain. Another key choice is the use of the WarehouseSystem class to keep track of the products in stock and generate alerts for warehouse staff when necessary. This allows for real-time updates to stock levels and ensures that the shelves are always well-stocked, improving customer experience.

A list of relationships between the classes:

  • Product and ShoppingBasket: The Product class is associated with the ShoppingBasket class through a one-to-many aggregation relationship, meaning that an instance of the ShoppingBasket class would contain multiple instances of the Product class. This relationship is also present between the Product class and the WarehouseSystem class.
  • Staff and WarehouseSystem: The Staff class is associated with the WarehouseSystem class through a one-to-many aggregation relationship, meaning that an instance of the WarehouseSystem class would contain multiple instances of the Staff class.
  • Customer and LoyaltyCard: The Customer class is associated with the LoyaltyCard class through an inheritance relationship, meaning that an instance of the LoyaltyCard class is a sub-class of the Customer class.
  • PointOfServiceSystem and ShoppingBasket, Receipt, PaymentProcessor, Customer, WarehouseSystem, and Staff: The PointOfServiceSystem class is associated with above mentioned classes through a composition relationship, meaning that an instance of the PointOfServiceSystem class would contain and use the other classes to coordinate the self-checkout process.
