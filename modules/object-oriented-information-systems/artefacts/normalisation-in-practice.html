Normalisation in practice

Given a set of database tables for a property lettings business, I have develop a table structure that conforms to third normal form (3NF).

The relationships between the tables are as follows:

  1. One-to-one relationship between Property and Owner, where the primary key of the Properties table (Property_ID) is a foreign key in the Owners table (Owner_ID). This means that each property has one owner and each owner can own one property.
  2. One-to-one relationship between Property and Client, where the primary key of the Properties table (Property_ID) is a foreign key in the Clients table (Client_ID). This means that each property has one client and each client can rent one property.
  3. One-to-many relationship between Property and Maintenance, where the primary key of the Properties table (Property_ID) is a foreign key in the Maintenance table (Property_ID). This means that each property can have multiple maintenance records and each maintenance record is associated with one property.
  4. One-to-one relationship between Maintenance and Tenant, where the primary key of the Maintenance table (Tenant_ID) is a foreign key in the Tenants table (Tenant_ID). This means that each maintenance record is associated with one tenant and each tenant can have one maintenance record.
  5. One-to-one relationship between Maintenance and Contractor, where the primary key of the Maintenance table (Contractor_ID) is a foreign key in the Contractors table (Contractor_ID). This means that each maintenance record is associated with one contractor and each contractor can have one maintenance record.

By having these relationships, the data is now well-organized and follows the third normal form (3NF) principles of no duplicated data, no partial dependency and no transitive dependency. This also makes it easy to maintain and update the data as well as making it more efficient for querying and reporting.

Module learning outcomes

  1. Using appropriate tools: Properly designing a table structure that conforms to 3NF requires understanding of database design principles and using tools such as ERD (Entity Relationship Diagram) to model the relationships between entities.
  2. Object-oriented information system: A database is an object-oriented information system, and designing the table structure in 3NF ensures that the database is structured in a way that follows the principles of object-oriented design, such as encapsulation and inheritance.
  3. Documenting: When designing a table structure that conforms to 3NF, it is important to document the design decisions made, such as the primary and foreign keys, and the relationships between tables. This documentation will be important for maintenance and future modifications of the system.

Regarding the module learning outcome, "Appraise and evaluate critically the concepts and principles of information systems," designing a table structure that conforms to 3NF relates to it in the following ways:

  1. Appraise: Understanding the concepts and principles of 3NF, such as the elimination of data redundancy and dependency, allows for the appraisal of existing table structures and identifying areas of improvement.
  2. Evaluate critically: By understanding the benefits and drawbacks of 3NF, it is possible to evaluate critically the design decisions made and weigh the trade-offs between normalizing the data and maintaining performance.
  3. Principles of information systems: 3NF is a fundamental principle of database design and is an important concept to understand when designing information systems. Understanding 3NF allows for the design of efficient and maintainable database systems.
